# CLAUDE.md — WOWVote 협업 규칙

이 파일은 Claude Code가 WOWVote 작업 시 따를 행동 규칙입니다.

---

## 🌟 가장 중요한 원칙

**이 프로젝트는 데모 앱이 아니라 WOWD.LAB의 실제 서비스 자산입니다.**

`vote.wowdlab.com`에 배포되어 시민 워크숍·세미나에서 실제 운영될 도구예요. 그래서 모든 코드는 다음을 만족해야 합니다:

1. **확장 가능해야 함** — A에서 멈추지 않고 B/C로 진화할 수 있어야 함
2. **운영 가능해야 함** — 진행자(테디 또는 외부 사용자)가 실제로 쓸 수 있어야 함
3. **재사용 가능해야 함** — 워크숍 참가자가 자기 환경에 셋업해서 학습할 수도 있어야 함

"빨리 동작하는 것"보다 "단단하고 확장 가능한 것"이 우선.

---

## 기본 응답 원칙

- 모든 응답은 **한국어**로
- 코드 주석은 한국어로 풍부하게
- 변수명·함수명은 영문 그대로, 설명만 한국어
- 거창한 사전 설명 없이 바로 작업으로 들어간다
- 작업 후 결과 요약은 간결하게 (불릿 3~5개)

- **한 Phase 끝낼 때마다 반드시 테스트 실행**
- 코드만 작성하고 끝내지 말 것
- 가능한 경우 실제 데이터로 동작 확인

### current_status.md 업데이트 의무

- "아마 이럴 거예요" 금지
- 환경변수, API 응답 구조, Supabase 정책 등 불확실하면 확인 요청
- 사용자의 의도가 모호하면 짧게 되묻기

## 코드 작성 규칙

### 일반
- 외부 빌드 도구 사용 금지 (Vanilla HTML/CSS/JS)
- 라이브러리는 CDN 사용 (npm install 불필요)
- 한 파일 300줄 이내 목표
- 모든 자바스크립트는 `defer`로 로드

### 명명 규칙
- 변수·함수: camelCase
- 클래스: PascalCase
- 파일명: kebab-case (예: `present-view.html`)
- DB 컬럼명: snake_case

### 주석
- 함수 위에 한국어로 "이 함수가 무엇을 하는지" 1~2줄
- 복잡한 로직 안에는 "왜 이렇게 하는지" 설명
- TODO·FIXME는 한국어로

### 에러 처리
- 모든 Supabase 호출은 try-catch
- 에러 메시지는 한국어로 사용자에게 친절히
- 콘솔 로그는 디버깅 끝나면 제거

## 보안 원칙 (절대 위반 금지)

### 1. service_role 키는 절대 클라이언트 노출 금지
- 모든 클라이언트 코드는 **anon public 키만** 사용
- service_role 키는 .env에도 넣지 말 것 (오늘은 안 씀)

### 2. .env 파일은 절대 git에 올라가지 않게
- `.gitignore`에 `.env` 포함 확인 (커밋 전 매번)
- `.env.example`만 git에 포함

### 3. XSS 방지
- 모든 사용자 입력은 `textContent` 또는 escape 함수로 렌더링
- `innerHTML`에 직접 사용자 입력 삽입 금지
- 예외: 본인이 만든 HTML 템플릿만 가능

### 4. SQL Injection 방지
- Supabase 라이브러리 사용 시 자동 방지됨
- 단, raw SQL 호출 시에는 반드시 파라미터 바인딩

### 5. RLS는 항상 켜둘 것
- 어떤 테이블도 RLS 끄지 말 것
- 정책으로 권한 제어

## Supabase 작업 시

### 스키마 변경
- 모든 스키마 변경은 SQL Editor에서 실행
- 변경 후 `supabase-migrations/` 폴더에 SQL 파일로 기록 (이력 관리)
- 변경 전 사용자에게 확인 받기

### 정책(Policy) 작업
- RLS 정책 변경은 보안에 직접 영향 → 사용자에게 확인 후 진행
- 정책 이름은 영문 소문자 + 언더스코어 (예: `select_all_questions`)

### 쿼리 최적화
- N+1 문제 방지: 한 번에 가져올 수 있으면 join 사용
- 정렬·필터는 DB에서 (클라이언트 정렬 최소화)
- 인덱스가 필요한 컬럼은 미리 만들기

---

## 확장성 원칙 (B/C 대비)

### 1. 데이터 모델은 처음부터 미래 대비
- `sessions` 테이블에 default 세션 하나만 쓰더라도, B에서 다중 세션 가능하게 설계
- `is_answered`, `is_hidden` 컬럼은 A에서 안 써도 미리 만들어둠

### 2. 코드는 "여러 세션" 가능하게
- 항상 `session_id`를 필터로 사용 (A에서는 default 자동 선택)
- 절대 "WHERE id IN (...)" 같이 하드코딩하지 말 것

### 3. UI는 확장 자리 미리
- 발표자 페이지(`/present`)는 A에서도 만들기 (B/C에서 정교화)
- 모더레이션 액션은 A에서는 안 보이지만 컴포넌트 자리는 둘 것

### 4. 환경변수로 분리
- `default_session_code = 'default'` 같은 값은 환경변수로
- B/C 갈 때 코드 수정 없이 환경변수만 바꿔서 동작

## 워크숍 참가자 학습 가치

이 레포는 워크숍 참가자들이 받아서 자기 Supabase로 셋업하는 학습 자료이기도 합니다.

### 학습 관점에서 지킬 것
- 코드는 읽기 쉽게 (난해한 트릭 금지)
- 주석은 "왜"에 집중 (코드만 봐도 "무엇"은 알 수 있음)
- README에 셋업 단계 명확히 (Clone → DB → .env → 실행)
- 참고할 만한 Claude Code 프롬프트 예시 제공

### 학습자가 막혔을 때 풀 수 있게
- 트러블슈팅 가이드 (.env 위치, RLS 정책 등)
- 자주 발생하는 에러와 해결책

---

## 사용자(테디)와의 소통 패턴

- 테디는 짧은 확인 응답을 선호 ("OK", "응", "좋아")
- 결정이 필요할 때는 **선택지 2~3개를 제시**하고 추천도 함께
- 한 번에 모든 걸 보여주지 말고 단계별로
- 코드 변경이 크면 변경 사항을 먼저 요약 → 동의 받고 실행
- "뾰족하게" 표현 — 핵심을 짧게

---

## 막혔을 때 행동 순서

1. 에러 메시지를 그대로 사용자에게 보여주고 원인 분석
2. 2~3개 해결 방안 제시
3. 사용자가 선택하면 실행
4. 실행 후 결과 보고

**추측으로 해결책을 시도하지 말 것.**
